Spring Security

一个重量级的安全管理框架：

  • 配置繁琐
  • 概念繁琐

优势：

  • 安全

在 SSM 里，因为配置繁琐，所以 Spring Security 使用较少。但是，Spring Boot 中为 Spring Security 做了自动化配置，所在 Boot 中，security 使用非常多。

获取登录用户信息

可以通过SecurityContextHolder来获取用户登录信息

@GetMapping("/hello")
public String sayHello(){
    SecurityContext context = SecurityContextHolder.getContext();
    Authentication authentication = context.getAuthentication();
    Object principal = authentication.getPrincipal();
    System.out.println("Principal = " + principal);
    new Thread(()  -> {
        Authentication auth = SecurityContextHolder.getContext().getAuthentication();
        System.out.println("auth.getPrincipal() = " + auth.getPrincipal());
    }).start();
    return "你好啊~~";
}

不过我们会发现，换一个线程就无法获取到信息了，这是因为spring security会把信息存到ThreadLocal中。

spring security究竟是怎么存用户信息的？

首先，服务端一定是将用户信息存到 HttpSession。

  1. 首先前端发起一个登录请求，这个请求会经过 Spring Security 一系列的过滤器，其中SecurityContextPersistenceFilter 过滤器，主要用来处理当前用户信息。当用户登录成功之后，会自动将用户信息存入到 SecurityContextHolder 中，SecurityContextHolder 的底层，就是 ThreadLocal。
  2. 在登录请求处理完毕后，响应数据给前端的时候，也会经过 SecurityContextPersistenceFilter 过滤器，此时，系统会自动将刚刚登录成功的用户信息从 SecurityContextHolder 中==取出来存入到 HttpSession 中==。
  3. 下一次请求来的时候，还是会经过 SecurityContextPersistenceFilter 过滤器，此时系统会自动从 HttpSession 中读取出当前登录成功的用户信息，并将之存入到 SecurityContextHolder 。

另，新版本已经换成了：SecurityContextPersistenceFilter，以下是文档信息

A javax.servlet.Filter that uses the SecurityContextRepository to obtain the SecurityContext and set it on the SecurityContextHolder. This is similar to SecurityContextPersistenceFilter except that the SecurityContextRepository.saveContext(SecurityContext, HttpServletRequest, HttpServletResponse) must be explicitly invoked to save the SecurityContext. This improves the efficiency and provides better flexibility by allowing different authentication mechanisms to choose individually if authentication should be persisted.
一个使用SecurityContextRepository来获取SecurityContext 并且将其设置在SecurityContextHolder的javax.servlet.Filter过滤器。除了saveContext必须明确引入保存SecurityContext的方法之外与SecurityContextPersistenceFilter 很相似。这提高了效率，并且通过允许分别选择不同认证方式（如果认证信息需要被保存）来提供更高的灵活性。

为什么要这么麻烦，直接存到session中不好吗?而且为什么每次登录时都要重新存取一次？

第一个问题，session中确实是可以取到用户信息的，像这样：

new Thread(()  -> {
    SecurityContext securityContext = (SecurityContext) session.getAttribute("SPRING_SECURITY_CONTEXT");
    User p = (User) securityContext.getAuthentication().getPrincipal();
    System.out.println("Principal = " + p);
    //这样在新的线程也是可以获取到的
}).start();

SecurityContext 只是提供了更便捷的方式来获取用户信息。

第二个问题，每次登录，用户信息都可能变化，所以需要及时保存用户信息。

现在可以通过spring security进行验证，那如果我想修改登录页面，要怎么做？

需要给特殊页面放行

自定义页面放行
springsecurity给页面放行有两种方式

  1. 放行的资源，不需要经过spring security过滤器链 （静态资源一般用这种）
  2. 经过spring security过滤器链，但是不拦截（如果一个接口想要匿名访问，一般用这种）

两种方式的实例

package com.hengixng.demo.config;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.config.annotation.web.configuration.WebSecurityCustomizer;
import org.springframework.security.web.SecurityFilterChain;

/**
 * @author hengxing
 * @version 1.0.0
 * @Description TODO
 * @createTime 2023年05月20日 15:02:00
 */
@Configuration
public class SecurityConfig {

    /**
     * 自定义页面放行，这是第一种
     * @Returns org.springframework.security.config.annotation.web.configuration.WebSecurityCustomizer
     * @Author hengxing
     * @Date 2023/5/20
     * @Time 15:05:29
     */
    @Bean
    WebSecurityCustomizer securityCustomizer(){
        return web -> web.ignoring().antMatchers("/login.html");//匿名类的lambda写法	
    }

    /**
     *  这是第二种
     * 自己配置过滤器链。里面的方法就是一条链接的一生~~
     * @Returns org.springframework.security.web.SecurityFilterChain
     * @param http  请求
     * @Author hengxing
     * @Date 2023/5/20
     * @Time 15:21:30
     */
    @Bean
    SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception{
        //开始认证
        http.authorizeRequests()
                .antMatchers("/login.html").anonymous()//路径为/login.html的请求可以匿名访问
                .anyRequest()//所有请求，类似shrio中的 /**
                .authenticated()//必须要认证后才能访问，类似shrio中的authc
                .and()
                .formLogin()//开始配置登录表单
                .loginPage("/login.html")//配置登录页面，如果访问了一个需要认证之后才能访问的页面，会自动跳转到这个页面来
                .loginProcessingUrl("/doLogin")//配置处理登录请求的接口，本质上就是配置过滤器的拦截规则，将来的登录请求就会在过滤器中被处理。
        //这个不需要你自己写一个doLogin接口
                .usernameParameter("username")//配置登录表单中用户名的key
                .passwordParameter("password")
                .defaultSuccessUrl("/hello")//登录成功后的跳转地址
                .failureUrl("/login.html")
                .and()
                .csrf().disable();//先关闭csrf防御机制，这个disable方法本质上就是从过滤器链上移除csrf过滤器
        return http.build();
    }
}

securityCustomizer()给我们设置了页面放行

securityFilterChain是我们自定义过滤器链的方法，规定了一个请求进来后如何进行验证

  • 相当于自己生成一个过滤器链，定义如何验证访问的请求
  • and()的作用就是把条件并列起来，你也可以重起一行，使用http对象调用。
  • csrf防御机制后面再介绍，先关闭

未登录信息返回

如果用户没有登录，就访问一个需要登录后才能访问的界面，我们之前的做法是将其跳转到登录页面，但如果我们是前后端分离的页面，这个事情交给前端自己处理就好了，我们只需要返回用户未登录的信息。

.exceptionHandling()
    //如果用户未登录就访问某一个页面，就会触发当前方法
.authenticationEntryPoint((request, response, authException) -> {
    response.setContentType("application/json;charset=utf-8");
    RespBean respBean = RespBean.error("尚未登录，请登录");
    String s = new ObjectMapper().writeValueAsString(respBean);
    response.getWriter().write(s);
  });

密码加密

springsecurity默认使用代理密码解码器DelegatingPasswordEncoder，源码中的注释如下

A password encoder that delegates to another PasswordEncoder based upon a prefixed identifier.

一个基于前缀标识的密码解码器，可以代理/委托任务给另一个密码解码器。

也就是说，这个密码解码器并没有实际的解码，只是根据不同前缀，把密码交给对应的解码器。所以spring security默认支持多密码策略。

我们可以在密码解密工厂类中看到如下代码

public static PasswordEncoder createDelegatingPasswordEncoder() {
		String encodingId = "bcrypt";
		Map<String, PasswordEncoder> encoders = new HashMap<>();
		encoders.put(encodingId, new BCryptPasswordEncoder());
		encoders.put("ldap", new org.springframework.security.crypto.password.LdapShaPasswordEncoder());
		encoders.put("MD4", new org.springframework.security.crypto.password.Md4PasswordEncoder());
		encoders.put("MD5", new org.springframework.security.crypto.password.MessageDigestPasswordEncoder("MD5"));
		encoders.put("noop", org.springframework.security.crypto.password.NoOpPasswordEncoder.getInstance());
		encoders.put("pbkdf2", new Pbkdf2PasswordEncoder());
		encoders.put("scrypt", new SCryptPasswordEncoder());
		encoders.put("SHA-1", new org.springframework.security.crypto.password.MessageDigestPasswordEncoder("SHA-1"));
		encoders.put("SHA-256",
				new org.springframework.security.crypto.password.MessageDigestPasswordEncoder("SHA-256"));
		encoders.put("sha256", new org.springframework.security.crypto.password.StandardPasswordEncoder());
		encoders.put("argon2", new Argon2PasswordEncoder());
		return new DelegatingPasswordEncoder(encodingId, encoders);
	}

这个方法的描述：

Creates a DelegatingPasswordEncoder with default mappings.

在默认映射情况下，创建一个代理密码解码器。

如果我们不需要这个代理，也可以自己注册一个bean，表示使用明文密码。

@Bean
PasswordEncoder passwordEncoder(){
    //表示使用明文密码
    return NoOpPasswordEncoder.getInstance();
    //表示使用BCrypt密码
    //return new BCryptPasswordEncoder();
}

那此时数据库的密码中就不需要写前缀了，因为现在已经不是使用代理解码器，而是明文。

密码升级

代理解码器在升级旧有密码时十分有用。可以快速重置新的密码策略

spring security可以帮我们升级密码，在用户登录时根据情况确定是否修改密码策略

我们只需要提供一个更新密码的服务即可：

让服务类实现UserDetailsPasswordService接口，实现内部更新密码的方法
当用户登录之后，系统会自动判断用户密码是否需要升级，依据是两个：

  1. 这个用户的密码是否是当前默认的方案，如果不是，就升级
  2. 同一种加密方案，也可升级（升级密码的强度）
@Override
public UserDetails updatePassword(UserDetails user, String newPassword) {
    User u = (User) user;
    u.setPassword(newPassword);
    userMapper.updatePassword(u);
    return null;
}

另外提供userMapper的更新密码方法即可。

void updatePassword(User u);
<update id="updatePassword">
    update t_user set password= #{password} where id = #{id};
</update>

这样，用户下次登录时，先使用旧有策略验证登录，登录成功后，系统就知道当前的密码了，直接更新为新的密码策略，再保存即可。

另外升级强度，用bcrypt做例子，默认的强度为10，可以看到密码中{bcrypt}$2a$10$最后的10，就是他的强度。我要将其修改为12。

我们先修改一下密码解码器：

@Bean
PasswordEncoder passwordEncoder(){
    String encodingId = "bcrypt";
    Map<String, PasswordEncoder> encoders = new HashMap<>();
    encoders.put(encodingId, new BCryptPasswordEncoder(12));//修改强度为12
    encoders.put("noop", org.springframework.security.crypto.password.NoOpPasswordEncoder.getInstance());
    return new DelegatingPasswordEncoder(encodingId, encoders);
}

注意这里的encodingId就是默认的密码策略。

Remember Me

这个功能可以让电脑保存登录信息，在用户关闭浏览器之后，再次访问该页面时不需要再次登录。

添加rememberMe功能很简单，加一个方法就行了。

默认情况下，所有的页面都可以通过rememberMe登录，本质上就是向过滤器链中添加了一个RememberMeAuthenticationFilter

@Bean
SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http.authorizeRequests()
        .anyRequest().authenticated()
        .and()
        .formLogin()
        .and()
        .rememberMe()//添加rememberme功能
        .and()
        .csrf().disable();
    return http.build();
}

如果我希望某些页面必须通过验证才能访问呢？比如支付界面，不论是否之前登录过，每次进入都要验证。

在过滤器链设置这个请求必须经过认证即可。

@Bean
SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http.authorizeRequests()
        .antMatchers("/admin").fullyAuthenticated()//只允许用户名密码登录验证访问
        .anyRequest().authenticated()
        .and()
        .formLogin()
        .and()
        .rememberMe()
        .and()
        .csrf().disable();
    return http.build();
}

服务器重启之后，rememberMe就失效了

要理解这个问题的原因，首先要了解rememberMe的工作原理。

开启 RememberMe 之后，登录成功后，服务端会响应一个 rm 字符串回来（在请求头的cookie中）：emhhbmdzYW46MTY1OTQwMzM3NzE0ODowNjczYzdlYWFlNjE5MWU1YzY1MDYyOWRhMWUwZWQ5Zg ，这是一个 base64 编码之后的字符串，解码之后，分为三部分：

hengxing:1685959542908:3e9c50c2365d64f2b377d1baa03cfa5c

  • 用户名
  • 时间戳
  • 加密字符串（根据用户名+用户密码+时间戳+key加密生成的字符串，不可解密）

以后每次请求的时候，都会自动携带上这个 Cookie，服务端收到 Cookie 之后，会解析出来用户名和时间戳，通过时间戳就能判断出 Cookie 是否已经过期。没有过期的话，根据用户名查询出用户密码，然后根据和用户名、用户密码、时间戳、key 进行加密，算出加密字符串，将加密字符串跟 Cookie 中的加密字符串进行比较。

从以上信息中，我们可以分析出一下两点：

  1. 该cookie两周后过期，时间戳转换后为：2023-06-05 18:05:42。写此文章时时间为2023年5月22日，刚好两周。
  2. 字符串中有一个变量key，如果key发生变化，我们重新计算的字符串也不相同。

问题找到了，是==由于key不同造成的cookie失效==，我们只需要保证每次的key都相同就行了。

实际上，如果我们不指定key，spring security会随机生成，所以你会发现每次重启后就失效。要想指定也很简单：

 @Bean
SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http.authorizeRequests()
        .antMatchers("/admin").fullyAuthenticated()//只允许用户名密码登录验证访问
        .antMatchers("/hello2").rememberMe()//只能通过rememberMe访问
        .anyRequest().authenticated()
        .and()
        .formLogin()
        .and()
        .rememberMe()
        .key("hengxing")//在这里！！！！！！
        .and()
        .csrf().disable();
    return http.build();
}

在rememberMe()方法后面加上key方法来指定

这样无论服务器如何重启，都不会失效。

会话管理

用最新的项目来测试，我们会发现，一个用户可以在多个设备上登录。

如何限制一个用户只能在一个设备登录。

@Bean
SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http.authorizeRequests()
        .anyRequest().authenticated()
        .and()
        .formLogin()
        .and()
        .sessionManagement()//session管理器
        .maximumSessions(1)//最大会话数量
        .and().and()
        .csrf().disable();
    return http.build();
}

再次登陆时，就会显示这样的提示

This session has been expired (possibly due to multiple concurrent logins being attempted as the same user).

实际上，这个操作只是帮我们注册了SessionManagementFilter这个过滤器，我们如果不添加这个属性，spring security就不注册这个过滤器。

默认情况下，达到最大会话数后，再次登录会挤掉上一次用户登录的信息。如果我们希望==达到最大会话数时阻止新用户登录==，就设置这个参数：

.maxSessionsPreventsLogin(true)//阻止用户登录，在session管理器的最大会话数量后面设置。

原理

在服务端，Spring Security 维护了一个会话注册表，所有登录上来的用户，都会注册到这个会注册表中，所谓的会话注册表，本质上是一个 Map 集合，Map 的 key 是当前用户对象，Map 的 value 是一个集合，这个集合中保存着这个用户的所有会话。Map<User,List<Session>>。

那么理论上，我们在用户登录后进行注销（保证同时登录的设备只有一个），再次登录，应该是可以登录的，但是我们试验发现不行。这是为什么？

当用户注销登录的时候，用户的 session 会被自动销毁，但是 Map 中的 List 集合中的 session 并不会自动移除。所以，应当在用户注销登录的时候，将 List 集合中，和用户对应的会话移除掉。

原来是spring security还不够智能，我们手动删除一下就可以了

@Bean
HttpSessionEventPublisher httpSessionEventPublisher(){
    return new HttpSessionEventPublisher();
}

这个publisher可以监听到用户的注销操作，并把它广播出去，spring security就可以监听到了。

一个小问题

我们返回使用账户密码登录的项目security03中，再次测试限制登录数量的功能（这里记得在设置中写上限制登录的参数，如下），发现又失效了。

http.sessionManagement().maximumSessions(1)
    .maxSessionsPreventsLogin(true);

这是因为spring security在验证用户的时候Map<User,List<Session>>，使用user做key。那么验证是否为同一用户时，就会使用user对象的equals()方法，但是我们并没有重写equals，用的是object的方法，比对地址。这样返回的当然是false啦，因为每次登录时，我们都是新建的对象。

==解决办法就是重写equals和hashcode方法。==

会话固定攻击与CSRF攻击

会话固定攻击

在登录网站后,网站会保存和我们的会话信息(保存在服务器中),将sessionId分配给我们,我们通过在请求时携带sessionId,服务器就知道,哦~原来是你.

这本来是很方便的,通过sessionId我们可以确定一个用户的会话是不是认证过的,而且即使浏览器重启后,我们也不会丢失会话信息,而是可以通过sessionId来还原会话,省去了输入密码认证的麻烦.

可是,sessionId既然我们能用,攻击者也能用.攻击者通过将自己的sessionId传给你,在你登录认证之后,用==认证后==的sessionId,进行一些敏感操作,比如将你的浏览记录发布在朋友圈~~

那我们如何防御?

这个问题的根源在 sessionid 不变，如果用户在未登录时拿到的是一个 sessionid，登录之后服务端给用户重新换一个 sessionid，就可以防止会话固定攻击了。

spring security中的防御主要体现在三方面:

首先就是请求地址中有; 请求会被直接拒绝。
另一方面就是响应的Set-Cookie字段中有HttpOnly属性，这种方式避免了通过XSS攻击来获取Cookie中的会话信息进而达成会话固定攻击。
第三点则是让 sessionid 变一下。既然问题是由于 sessionid 不变导致的，那我就让 sessionid 变一下。

我们可以在过滤器链设置中,设置session

@Bean
SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http.authorizeRequests()
        .antMatchers("login").anonymous()
        .anyRequest()
        .authenticated()
        .and().formLogin()
        .and()
        .sessionManagement()
        .sessionFixation().migrateSession();//设置migrateSession,登录后会变动sessionid
    return http.build();
}
  1. migrateSession表示在登录成功之后，创建一个新的会话，然后讲旧的 session 中的信息复制到新的 session 中，默认即此。
  2. none表示不做任何事情，继续使用旧的 session。
  3. changeSessionId表示 session 不变，但是会修改 sessionid，这实际上用到了 Servlet 容器提供
    的防御会话固定攻击。
  4. newSession表示登录后创建一个新的 session。

自己实现一个PermissionEvaluator的实现类，重写hasPermission方法，调用ant风格的路径匹配来对比权限项

package com.hengxing.demo.config;

import org.springframework.security.access.PermissionEvaluator;
import org.springframework.security.core.Authentication;
import org.springframework.security.core.GrantedAuthority;
import org.springframework.stereotype.Component;
import org.springframework.util.AntPathMatcher;

import java.io.Serializable;
import java.util.Collection;

@Component
public class MyPermissionEvaluator implements PermissionEvaluator {

    AntPathMatcher antPathMatcher = new AntPathMatcher();

    /**
     * 返回根据antPath判断的结果
     * @Param authentication 用户的权限信息
	 * @Param targetDomainObject
	 * @Param permission 需要判断的权限项
     */
    @Override
    public boolean hasPermission(Authentication authentication, Object targetDomainObject, Object permission) {
        Collection<? extends GrantedAuthority> authorities = authentication.getAuthorities();
        for (GrantedAuthority authority : authorities) {
            if (antPathMatcher.match(authority.getAuthority(), (String) permission)) return true;
        }
        return false;
    }
}

其次设置解析JWT信息的过滤器链

package com.example.jwtdemo.config;

import io.jsonwebtoken.*;
import org.springframework.security.authentication.UsernamePasswordAuthenticationToken;
import org.springframework.security.core.GrantedAuthority;
import org.springframework.security.core.authority.AuthorityUtils;
import org.springframework.security.core.context.SecurityContextHolder;
import org.springframework.stereotype.Component;
import org.springframework.web.filter.GenericFilterBean;

import javax.servlet.FilterChain;
import javax.servlet.ServletException;
import javax.servlet.ServletRequest;
import javax.servlet.ServletResponse;
import javax.servlet.http.HttpServletRequest;
import java.io.IOException;
import java.util.List;

/**
 * 自定义过滤器，在这个过滤器中，解析每一个请求所携带的Jwt字符串
 * jwt字符串可以放在请求头，也可以放在请求参数中
 */
@Component
public class JwtFilter extends GenericFilterBean {

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) throws IOException, ServletException {
        HttpServletRequest httpServletRequest = (HttpServletRequest) request;
        //在这里要排除掉登录的请求，因为登录请求中肯定不会有jwt信息啊
        if ("/login".equals(httpServletRequest.getRequestURI()) && "POST".equalsIgnoreCase(httpServletRequest.getMethod())){
            //说明是一个登录请求,直接执行后面的操作即可
            chain.doFilter(request,response);
            return;
        }

        String token = httpServletRequest.getHeader("Authorization");
        if (token == null || "".equals(token)) {
            //请求头没有jwt字符串，尝试在请求参数里寻找
            token = httpServletRequest.getParameter("token");
            if (token == null || "".equals(token)) {
                response.getWriter().write("authorization failed, unable to find Jwt info.");
                return;
            }
        }
        try {
            /*
            * 解析Jwt令牌
            * token.replace("Bearer ","") 是为了把请求中发过来的前缀去掉
            * parseClaimsJws()这个方法中的jws是JWS (JWT with signature) 不要写错成 jwt
            * */
            Jws<Claims> claimsJws = Jwts.parserBuilder().setSigningKey(SpringConfig.KEY).build().parseClaimsJws(token.replace("Bearer ", ""));
            //获取用户名
            String username = claimsJws.getBody().getSubject();
            //获取用户权限
            String authority = claimsJws.getBody().get("authority", String.class);
            List<GrantedAuthority> grantedAuthorities = AuthorityUtils.commaSeparatedStringToAuthorityList(authority);
            //这就是我们恢复出来的用户信息
            UsernamePasswordAuthenticationToken authenticationToken = new UsernamePasswordAuthenticationToken(username, null, grantedAuthorities);
            /*spring security框架后续执行的过程中，凡是需要用到当前用户身份的地方，都是使用
            现在存储到context中
             */
            SecurityContextHolder.getContext().setAuthentication(authenticationToken);

            //jwt信息解析完成，继续执行后面的过滤器
            chain.doFilter(request,response);
        } catch (Exception e) {
            response.getWriter().write("authorization failed, something wrong when parse Jwt info.");
        }

    }
}

我来分布解析如何生成这个过滤器链

  1. 继承通用过滤器 extends GenericFilterBean

  2. 重写其中的doFilter方法

    HttpServletRequest httpServletRequest = (HttpServletRequest) request;
//在这里要排除掉登录的请求，因为登录请求中肯定不会有jwt信息啊
if ("/login".equals(httpServletRequest.getRequestURI()) && "POST".equalsIgnoreCase(httpServletRequest.getMethod())){
    //说明是一个登录请求,直接执行后面的操作即可
    chain.doFilter(request,response);
    return;
}

    这里第一步的强转时为了使用HttpServletRequest中的getHeader()方法，不然我们没法获取请求头信息

    第二部分排除登录的请求，登录时不需要验证jwt

    String token = httpServletRequest.getHeader("Authorization");

    //请求头没有jwt字符串，尝试在请求参数里寻找
    token = httpServletRequest.getParameter("token");
    if (token == null || "".equals(token)) {
        response.getWriter().write("authorization failed, unable to find Jwt info.");
        return;
    }
}

    分步解析：

    1. 解析Jws字符串

      Jwts.parserBuilder().setSigningKey(SpringConfig.KEY).build().parseClaimsJws(token.replace("Bearer ", ""))

      这句是重点，我们要设置一个签名，就用我们之前在另一个类中设置的KEY，解析时使用的方法是parseClaimsJws，其中的jws是JWS (JWT with signature) 不要写错成 jwt。

      token.replace("Bearer ","")是为了把请求中发过来的前缀去掉。我们发送jwt字符串时会发现，headers中的Authorization信息为Bearer eyJhbGciOiJIUzI1NiJ9.eyJhdXRob3JpdHkiOiJ1c2VyIn0.XbXtmIqMYSLTuyxDK-GvzX1ytKUq0NZt13EKRMSzMt8，前面多了Bearer 需要去掉

    2. 获取Jwt中的信息

      //获取用户名
String username = claimsJws.getBody().getSubject();
//获取用户权限
String authority = claimsJws.getBody().get("authority", String.class);
List<GrantedAuthority> grantedAuthorities = AuthorityUtils.commaSeparatedStringToAuthorityList(authority);

      AuthorityUtils是spring security提供的方法，将用逗号分隔开的权限字符串转为权限List

    3. 还原用户认证信息，Jwt就是一个认证信息的字符串，现在我们要还原一个用户信息出来

      UsernamePasswordAuthenticationToken authenticationToken = new UsernamePasswordAuthenticationToken(username, null, grantedAuthorities);

    4. 把当前的用户信息设置为jwt的用户信息

      SecurityContextHolder.getContext().setAuthentication(authenticationToken);

    5. 解析完成后，继续执行后面的执行链

      chain.doFilter(request,response);

  3. 注册jwtFilter到执行器链中去

    @Autowired
JwtFilter jwtFilter;

@Bean
SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    //将jwt过滤器添加到过滤器链中去,在UsernamePasswordAuthenticationFilter前添加，也就是在用户名密码认证之前。
    http.addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class);

    //。。。省略
    return http.build();
}
